## Context

本次变更不是单纯补一份安装说明，而是要把 GT4 Web 从本地开发形态整理为可在目标麒麟 Linux 服务器上长期稳定运行的生产部署方案。项目本身是 pnpm monorepo，前端为 Vite 构建产物，后端为 Node.js + Fastify + Socket.IO，同时依赖 Redis 与 PostgreSQL。部署设计需要回答以下关键问题：前端与后端如何拆分发布，Nginx 如何作为统一入口代理 HTTP 和 WebSocket，请求如何在生产环境中路由，Node.js 进程如何守护，配置与日志如何管理，以及上线、回滚、巡检如何标准化。

目标服务器已知条件为 Kylin Linux Advanced Server V11，内核 6.6.x，具备固定 IP。设计方案需要优先采用 Linux 原生、维护成本低的组件组合，避免引入不必要的复杂编排系统。

## Goals / Non-Goals

### Goals

- 形成适用于目标麒麟 Linux 服务器的生产部署拓扑与实施步骤。
- 明确生产环境所需软件清单、版本选择原则和安装顺序。
- 定义前端静态资源发布、Nginx 站点配置、Node.js 后端常驻运行与自启动方案。
- 定义 Redis、PostgreSQL、环境变量、日志目录、部署目录和权限模型。
- 定义生产验证、故障恢复、滚动更新与回滚的执行方式。

### Non-Goals

- 不在本次设计中引入 Kubernetes、Docker Swarm 等容器编排平台。
- 不在本次设计中重构现有前后端业务逻辑。
- 不覆盖跨多台服务器的高可用集群方案，仅覆盖单机生产部署基线。

## Decisions

### 1. 部署拓扑采用“单机 Nginx + Node.js + Redis + PostgreSQL”模式

决策：服务器作为单机部署节点，Nginx 暴露 80/443 端口，前端构建产物由 Nginx 直接提供，/api 和 /socket.io 反向代理到本机 Node.js 服务；Redis 与 PostgreSQL 优先本机部署，或在文档中明确支持切换到独立主机。

理由：

- 与当前项目架构天然匹配，不需要改动接口路径设计。
- 目标环境是固定服务器，优先保证可维护性与上线确定性。
- Nginx 处理静态资源和反向代理比直接让 Node.js 暴露公网更稳妥。
- WebSocket 代理场景已知，Nginx 可统一处理升级头与超时配置。

备选方案：

- 由 Node.js 同时提供静态资源和 API。未采用，因为会增加后端职责，也不利于静态缓存与统一入口治理。
- 使用容器化部署。未采用，因为当前需求优先是可落地、低门槛的生产交付，而非容器平台治理。

### 2. 采用 systemd 守护 Node.js 后端进程

决策：后端服务使用 systemd unit 常驻运行，而不是依赖手工 nohup 或交互式终端。

理由：

- systemd 是 Linux 原生能力，适配麒麟服务器运维习惯。
- 支持开机自启、失败自动重启、统一日志、资源限制和状态查询。
- 减少对第三方进程管理器的额外依赖。

备选方案：

- PM2。可用但不是首选，因为多引入一层 Node.js 生态运维工具，长期维护收益不明显。

### 3. 构建与运行分离，生产服务器只承担最小必要的构建职责

决策：设计优先推荐“构建机产物交付到服务器”的模式，同时保留“服务器本机构建”的备用流程。最终运行目录应只包含前端 dist、后端构建结果、生产依赖、环境文件和日志目录。

理由：

- 构建与运行分离更利于发布稳定性和问题定位。
- 生产服务器减少编译工具链依赖，有利于降低变更面。
- 当现场网络或制品库条件受限时，仍可切换到服务器本机构建。

备选方案：

- 始终在目标服务器执行完整 pnpm install 与打包。未作为首选，因为会放大网络、锁文件、Node 版本和编译工具差异带来的不确定性。

### 4. 目录结构采用“版本目录 + current 软链接 + shared 配置目录”

决策：生产目录规划为 /opt/gt4_web/releases/<timestamp>、/opt/gt4_web/current、/opt/gt4_web/shared 等结构，配置文件、日志和上传类持久化内容放在 shared 或独立路径下。

理由：

- 便于原子切换版本与快速回滚。
- 避免新版本发布覆盖旧产物。
- 运维人员可以清晰区分代码、配置、日志和运行目录。

备选方案：

- 所有文件直接覆盖到单一目录。未采用，因为回滚困难，且容易混入历史残留文件。

### 5. Nginx 统一入口并显式支持 Socket.IO 代理

决策：Nginx 配置中区分静态站点、/api 代理、/socket.io WebSocket 代理，明确设置 proxy_http_version、Upgrade、Connection、Host、X-Forwarded-\* 等头部。

理由：

- 当前项目同时存在 REST API 与实时推送，两者都需要经过统一入口。
- WebSocket 代理若未显式配置升级头，生产环境将出现连接成功率低或断连问题。
- 统一入口便于后续增加 TLS、访问控制、限流与审计。

### 6. 环境配置采用 .env 文件 + systemd EnvironmentFile 注入

决策：后端运行所需的数据库、Redis、端口、运行模式等配置通过独立环境文件管理，并由 systemd 注入；前端若存在构建期变量，则在构建阶段通过对应环境文件固化。

理由：

- 保持代码与环境配置分离。
- 便于同一套代码在开发、测试、生产之间切换。
- 便于运维对敏感参数进行独立授权和备份。

### 7. 部署文档必须包含验收、巡检与回滚流程

决策：本次设计不仅描述安装步骤，还要求交付可执行的上线后验收清单、健康检查命令、日志查看方式、故障排查入口和回滚步骤。

理由：

- 生产部署的主要风险不在“装起来”，而在“稳定运行并可恢复”。
- 当前用户明确要求详细部署方案，因此必须覆盖运维闭环。

## Risks / Trade-offs

- 风险：目标麒麟 Linux 软件仓库中的 Node.js 版本过旧，无法满足当前项目依赖。
  缓解：在方案中优先推荐安装 LTS 版本 Node.js，并给出通过官方二进制包或受控源安装的路径，部署前先执行 node -v 与 pnpm -v 验证。

- 风险：Nginx 未正确代理 Socket.IO 升级头，导致实时通信异常。
  缓解：在部署方案中提供完整的 location /socket.io 配置，并将浏览器连接测试纳入验收步骤。

- 风险：生产服务器上直接构建失败，原因包括网络、原生模块编译工具、锁文件不一致。
  缓解：首选离线制品发布模式，并把服务器本机构建作为备用方案，提前定义两条流程。

- 风险：Redis 或 PostgreSQL 服务未就绪会导致 Node.js 后端启动失败。
  缓解：在 systemd 依赖和部署验收中加入数据库与 Redis 连通性检查，必要时将后端配置为启动前探测依赖。

- 风险：单机部署下任一核心组件故障都会影响整体可用性。
  缓解：明确该方案是单机生产基线，后续如需高可用，应在独立变更中引入主备或集群设计。

## Migration Plan

1. 在目标服务器上创建运行用户、目录结构与必要权限。
2. 安装并验证 Nginx、Node.js、pnpm、Redis、PostgreSQL 以及常用系统工具。
3. 准备数据库与 Redis 连接信息，创建生产环境变量文件。
4. 生成前端与后端生产制品，发布到 releases 目录。
5. 配置 Nginx 站点与 systemd 服务，完成开机自启。
6. 通过 current 软链接切换到新版本并启动服务。
7. 执行健康检查、API 检查、WebSocket 检查与日志巡检。
8. 若异常，回切 current 到上一版本并重启 Nginx / backend 服务。

## Open Questions

- Redis 与 PostgreSQL 是否必须部署在同一台目标服务器，还是允许接入现有独立实例。
- 生产环境是否需要 HTTPS 证书，以及证书来源是内网 CA 还是公网证书。
- 是否需要为现场环境补充离线安装包与离线升级说明。
- 后端是否需要额外引入 logrotate 或 journald 持久化策略，以控制长期日志体积。

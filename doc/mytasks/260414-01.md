# GT4 Web 在麒麟 Linux 服务器上的生产部署方案

## 1. 目标与边界

目标是在 140.32.1.191 这台 Kylin Linux Advanced Server V11 服务器上，以 Nginx 作为统一入口运行 GT4 Web：

- 前端 Vue + Vite 构建产物由 Nginx 直接托管。
- `/api` 和 `/socket.io` 由 Nginx 反向代理到本机 Node.js 后端。
- 后端以 systemd 服务方式常驻运行。
- Redis 与 PostgreSQL 可以部署在本机，也可以连接现有远端实例。

本方案优先给出单机生产基线，同时兼容当前仓库已存在的远端依赖模式。

## 2. 与当前仓库一致的运行事实

在制定部署方案前，需要先固定当前项目的实际运行约束：

- 项目是 pnpm monorepo，根目录要求 Node.js >= 20、pnpm >= 8。
- 根构建命令是 `pnpm run build`，会依次构建 `packages/shared`、`backend`、`frontend`。
- 后端生产启动命令是 `node dist/index.js`。
- 后端默认监听 `0.0.0.0:5001`。
- 健康检查接口是 `GET /api/health`。
- 开发期前端代理路径就是 `/api` 和 `/socket.io`，因此生产期沿用相同路径最稳妥。
- 前端静态构建输出目录是 `frontend/dist`。
- 当前代码已支持通过环境变量覆盖 Redis 地址、端口、密码和 DB 编号。
- 当前代码已调整为仅在非生产环境启动模拟数据生成器，生产环境只消费真实 Redis 数据。

## 3. 推荐生产拓扑

### 3.1 推荐拓扑

```text
浏览器
  -> http://140.32.1.191/
  -> Nginx :80 / :443
	 -> 静态文件: /opt/gt4_web/current/frontend/dist
	 -> /api        -> http://127.0.0.1:5001
	 -> /socket.io  -> http://127.0.0.1:5001

Node.js backend :5001
  -> PostgreSQL（本机或远端）
  -> Redis（本机或远端）
```

### 3.2 关于 Redis / PostgreSQL 的两种部署方式

建议先在部署前确认到底采用哪种依赖模式：

1. 应用与数据库分离
   当前仓库示例环境中，PostgreSQL 已经指向 140.32.1.192。若现场已有现成数据库和 Redis，140.32.1.191 只部署 Nginx + Node.js 即可。

2. 全部部署在 140.32.1.191 本机
   如果现场没有独立 Redis / PostgreSQL 服务，则需要在这台麒麟服务器上同时安装并初始化 Redis 与 PostgreSQL。

如果没有特别要求，建议先采用“应用部署在 140.32.1.191，数据库与 Redis 沿用现有服务”的方式，风险最低。

## 4. 目标服务器需要安装的软件

### 4.1 必装软件

- Nginx
- Node.js 20 LTS
- pnpm（通过 Corepack 管理）
- Git
- tar
- curl
- unzip 或 xz-utils
- systemd（系统自带）

### 4.2 按需安装的软件

- Redis
- PostgreSQL
- lsof / net-tools / ss 所在工具包
- firewalld 管理工具
- logrotate

### 4.3 为什么不建议直接依赖系统仓库的 Node.js

麒麟 Linux 的系统仓库中的 Node.js 版本不一定能满足本仓库的 Node.js >= 20 约束，因此建议直接安装官方 Node.js 20 LTS 二进制包，这样最稳定。

## 5. 服务器初始化

以下命令默认使用 root 执行。

### 5.1 创建运行用户和目录

```bash
useradd --system --create-home --home-dir /opt/gt4_web --shell /sbin/nologin gt4web

mkdir -p /opt/gt4_web/releases
mkdir -p /opt/gt4_web/shared
mkdir -p /opt/gt4_web/shared/logs
mkdir -p /opt/gt4_web/shared/run

chown -R gt4web:gt4web /opt/gt4_web
chmod 750 /opt/gt4_web
chmod 750 /opt/gt4_web/shared
chmod 750 /opt/gt4_web/shared/logs
```

### 5.2 安装基础工具

麒麟服务器通常可以使用 `dnf` 或 `yum`。下面以 `dnf` 为例；如果系统只有 `yum`，直接把 `dnf` 替换成 `yum` 即可。

```bash
dnf install -y git curl tar xz nginx
```

如果你计划把 Redis / PostgreSQL 也部署在本机，再额外安装：

```bash
dnf install -y redis postgresql postgresql-server
```

如果 PostgreSQL 包名和你系统仓库中的名称不一致，以本机可用包名为准。

## 6. 安装 Node.js 20 与 pnpm

### 6.1 安装 Node.js 20 LTS

```bash
cd /usr/local/src
curl -LO https://nodejs.org/dist/v20.19.0/node-v20.19.0-linux-x64.tar.xz
tar -xJf node-v20.19.0-linux-x64.tar.xz
ln -sfn /usr/local/src/node-v20.19.0-linux-x64 /usr/local/node
ln -sfn /usr/local/node/bin/node /usr/local/bin/node
ln -sfn /usr/local/node/bin/npm /usr/local/bin/npm
ln -sfn /usr/local/node/bin/npx /usr/local/bin/npx

node -v
```

如果你希望安装更新的 Node 20 小版本，可替换下载链接，但必须保持主版本为 20。

### 6.2 安装 pnpm

```bash
corepack enable
corepack prepare pnpm@10.30.3 --activate
pnpm -v
```

这里使用仓库 `packageManager` 中已经声明的 pnpm 版本，避免服务器与开发机不一致。

## 7. Nginx、Redis、PostgreSQL 的准备

### 7.1 Nginx 开机自启

```bash
systemctl enable nginx
systemctl start nginx
systemctl status nginx
```

### 7.2 如果 Redis 部署在本机

```bash
systemctl enable redis
systemctl start redis
systemctl status redis
```

### 7.3 如果 PostgreSQL 部署在本机

首次初始化数据库目录后启动：

```bash
postgresql-setup --initdb || true
systemctl enable postgresql
systemctl start postgresql
systemctl status postgresql
```

不同发行版命令可能略有差异，如果 `postgresql-setup` 不存在，则按系统提供的初始化脚本执行。

## 8. 部署目录规划

建议目录结构如下：

```text
/opt/gt4_web/
├── current -> /opt/gt4_web/releases/20260414-01
├── releases/
│   ├── 20260414-01/
│   └── 20260414-02/
└── shared/
	├── backend.env
	├── logs/
	└── run/
```

说明：

- `releases/<版本号>`：每次发布一个独立目录。
- `current`：当前在线版本软链接。
- `shared/backend.env`：生产环境变量。
- `shared/logs`：Nginx 或业务日志补充目录。

## 9. 当前仓库最适合的发布方式

### 9.1 推荐方案

当前仓库是 workspace 结构，后端运行还依赖 workspace 安装结果，因此现阶段最稳妥的方案是：

1. 把完整仓库源码发布到服务器的某个 release 目录。
2. 在 release 目录执行 `pnpm install --frozen-lockfile`。
3. 在 release 目录执行 `pnpm run build`。
4. 用 `current` 软链接切换版本。

这是当前仓库结构下最实用、最不容易踩坑的生产方案。

### 9.2 备用方案

如果未来要做“构建机产物交付、服务器不编译”的方案，需要再补充：

- 后端独立打包脚本
- 生产依赖裁剪方案
- workspace 内共享包的独立发布策略

当前仓库还没有这些配套，因此不建议第一版部署就走纯制品方式。

## 10. 发布步骤

以下步骤假设你在本地开发机已经把代码推送到 Git 仓库，服务器可以通过 Git 拉取代码。

### 10.1 创建新版本目录并拉取代码

```bash
export RELEASE_ID=$(date +%Y%m%d-%H%M%S)
mkdir -p /opt/gt4_web/releases/$RELEASE_ID
cd /opt/gt4_web/releases/$RELEASE_ID

git clone -b develop <你的仓库地址> .
```

如果服务器无法直接访问 Git，也可以在本地打包后通过 `scp` 上传，再在 release 目录解压。

### 10.2 安装依赖并构建

```bash
cd /opt/gt4_web/releases/$RELEASE_ID
pnpm install --frozen-lockfile
pnpm run build
```

构建成功后应至少确认两个目录存在：

- `frontend/dist`
- `backend/dist`

## 11. 生产环境变量

创建文件 `/opt/gt4_web/shared/backend.env`：

```ini
NODE_ENV=production
PORT=5001
FRONTEND_ORIGIN=http://140.32.1.191

DATABASE_URL=postgresql://username:password@db-host:5432/dbname?schema=public

REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_PASSWORD=
REDIS_DB=0
```

如果 PostgreSQL 和 Redis 仍然使用现有远端服务，请把 `db-host` 与 `REDIS_HOST` 改成对应实际地址，例如 140.32.1.192。

然后执行：

```bash
chown gt4web:gt4web /opt/gt4_web/shared/backend.env
chmod 640 /opt/gt4_web/shared/backend.env
```

## 12. systemd 配置 Node.js 后端服务

创建 `/etc/systemd/system/gt4-web-backend.service`：

```ini
[Unit]
Description=GT4 Web Backend Service
After=network.target
Wants=network.target

[Service]
Type=simple
User=gt4web
Group=gt4web
WorkingDirectory=/opt/gt4_web/current/backend
EnvironmentFile=/opt/gt4_web/shared/backend.env
ExecStart=/usr/local/bin/node dist/index.js
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

使其生效：

```bash
systemctl daemon-reload
systemctl enable gt4-web-backend
```

## 13. Nginx 配置

创建 `/etc/nginx/conf.d/gt4-web.conf`：

```nginx
server {
	listen 80;
	server_name 140.32.1.191;

	root /opt/gt4_web/current/frontend/dist;
	index index.html;

	access_log /var/log/nginx/gt4-web.access.log;
	error_log /var/log/nginx/gt4-web.error.log warn;

	location / {
		try_files $uri $uri/ /index.html;
	}

	location /api/ {
		proxy_pass http://127.0.0.1:5001;
		proxy_http_version 1.1;
		proxy_set_header Host $host;
		proxy_set_header X-Real-IP $remote_addr;
		proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
		proxy_set_header X-Forwarded-Proto $scheme;
	}

	location /socket.io/ {
		proxy_pass http://127.0.0.1:5001;
		proxy_http_version 1.1;
		proxy_set_header Upgrade $http_upgrade;
		proxy_set_header Connection "upgrade";
		proxy_set_header Host $host;
		proxy_set_header X-Real-IP $remote_addr;
		proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
		proxy_set_header X-Forwarded-Proto $scheme;
		proxy_read_timeout 600s;
		proxy_send_timeout 600s;
	}
}
```

校验并加载配置：

```bash
nginx -t
systemctl reload nginx
```

如果后续启用 HTTPS，可以在此基础上再加 443 server 块和证书配置。

## 14. 切换版本并启动服务

```bash
ln -sfn /opt/gt4_web/releases/$RELEASE_ID /opt/gt4_web/current
chown -h gt4web:gt4web /opt/gt4_web/current

systemctl restart gt4-web-backend
systemctl status gt4-web-backend

nginx -t
systemctl reload nginx
```

## 15. 防火墙与端口策略

建议公网只开放：

- 80
- 443（如果启用 HTTPS）

不建议对外开放：

- 5001
- 6379
- 5432

如果系统启用了 firewalld，可执行：

```bash
firewall-cmd --permanent --add-service=http
firewall-cmd --permanent --add-service=https
firewall-cmd --reload
```

## 16. 上线验收步骤

### 16.1 服务状态检查

```bash
systemctl status nginx
systemctl status gt4-web-backend
```

如果 Redis / PostgreSQL 在本机，再检查：

```bash
systemctl status redis
systemctl status postgresql
```

### 16.2 端口监听检查

```bash
ss -lntp | grep -E ':80|:443|:5001|:6379|:5432'
```

预期：

- Nginx 监听 80 或 443
- Node.js 监听 5001，但只绑定给本机或内网策略允许的地址

### 16.3 后端健康检查

```bash
curl http://127.0.0.1:5001/api/health
curl http://140.32.1.191/api/health
```

第二条命令能成功，说明 Nginx 反向代理和后端服务都正常。

### 16.4 前端页面检查

浏览器访问：

```text
http://140.32.1.191/
```

检查以下内容：

- 页面能正常打开
- 前端刷新后不会 404
- 访问页面时 `/api` 请求能正常返回
- 实时页面的 `/socket.io` 连接能建立成功

## 17. 日常运维与排障

### 17.1 查看后端日志

```bash
journalctl -u gt4-web-backend -f
```

### 17.2 查看 Nginx 日志

```bash
tail -f /var/log/nginx/gt4-web.access.log
tail -f /var/log/nginx/gt4-web.error.log
```

### 17.3 常见排障顺序

1. 先看 `systemctl status gt4-web-backend`
2. 再看 `journalctl -u gt4-web-backend -n 200`
3. 再看 `nginx -t` 和 Nginx error log
4. 再看 `curl http://127.0.0.1:5001/api/health`
5. 再看 Redis / PostgreSQL 连通性

### 17.4 Redis / PostgreSQL 连通性检查

如果 Redis 在本机：

```bash
redis-cli -h 127.0.0.1 -p 6379 ping
```

如果 PostgreSQL 在本机：

```bash
psql -h 127.0.0.1 -U <dbuser> -d <dbname> -c 'select 1;'
```

如果数据库和 Redis 在远端，把 `127.0.0.1` 改成对应实际地址。

## 18. 升级与回滚

### 18.1 标准升级流程

1. 创建新的 release 目录。
2. 上传或拉取新代码。
3. 执行 `pnpm install --frozen-lockfile`。
4. 执行 `pnpm run build`。
5. 执行 `ln -sfn` 切换 `current`。
6. 重启 `gt4-web-backend`。
7. 重载 Nginx。
8. 重新执行健康检查。

### 18.2 快速回滚流程

假设上一版本目录是 `/opt/gt4_web/releases/20260414-previous`：

```bash
ln -sfn /opt/gt4_web/releases/20260414-previous /opt/gt4_web/current
systemctl restart gt4-web-backend
systemctl reload nginx

curl http://127.0.0.1:5001/api/health
curl http://140.32.1.191/api/health
```

只要 release 目录完整保留，回滚就只需要切换软链接并重启服务。

## 19. 最终建议

第一版上线建议按下面顺序推进：

1. 先确认 Redis 和 PostgreSQL 是继续使用 140.32.1.192，还是迁移到 140.32.1.191 本机。
2. 先完成 Node.js 20、pnpm、Nginx 的安装。
3. 按本文创建目录、环境变量和 systemd 服务。
4. 先在服务器做一次完整 `pnpm install --frozen-lockfile && pnpm run build`。
5. 通过 `/api/health` 和浏览器联调确认无误后，再考虑 HTTPS、离线包和更规范的制品发布流程。

这样做的优点是：

- 与当前仓库结构最匹配。
- 不需要第一天就引入复杂发布系统。
- 可以先把生产环境稳定跑起来，再逐步优化自动化和高可用。
